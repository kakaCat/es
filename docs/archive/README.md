## ⚠️ 归档注意事项

### 何时归档文档？

文档满足以下条件之一时应该归档：
- ✅ 阶段性任务完成，相关报告文档完成使命
- ✅ 技术方案已过时，不再使用
- ✅ 需求已实现或取消
- ✅ 临时文档（如问题排查记录）问题已解决
- ✅ 根目录文档过多，需要精简

### 归档不等于删除

- ✅ **归档**: 移动到 archive/ 目录，保留完整历史
- ❌ **删除**: 物理删除文件，无法恢复
- 📦 **_old/**: 用于保留即将删除的代码和配置，定期清理

### 归档命名规范

```
YYYY-MM-DD-{event-name}/
│
├── README.md           # 归档说明（必需）
└── *.md                # 归档的文档
```

**示例**:
- `2025-12-01-project-cleanup/` - 项目整理
- `2025-11-15-auth-implementation/` - 认证系统实现
- `2025-10-20-architecture-redesign/` - 架构重设计

## 💡 归档最佳实践

1. **及时归档**: 项目阶段完成后立即归档，不要让根目录积累过多文档
2. **完整 README**: 每个归档目录必须有详细的 README.md
3. **保留链接**: 归档时更新其他文档中的引用链接
4. **定期审查**: 每季度审查一次，确定是否有新文档需要归档
5. **避免重复**: 归档前检查是否已存在相似文档
